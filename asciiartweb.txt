# ASCII Art Web Application - Complete Breakdown

Let me walk you through this entire project like I'm sitting next to you at a whiteboard. By the end, you'll understand every decision, every line, and be able to build something similar from scratch.

---

## The Big Picture

This is a **web application** written in Go that:
1. Shows a webpage with a form (text input + banner style selector)
2. User submits text, the server converts it to ASCII art
3. The result displays on the same page

Think of it like a restaurant: the user places an order (form submission), the kitchen (server) prepares the food (ASCII art), and serves it back on a plate (the same webpage).

## Part 1: Package and Imports

```go
package main
```
Every Go program starts here. `package main` means "this is an executable program, not a library." Go looks for `package main` and a `main()` function as the entry point — like how a car needs a key in the ignition.

```go
import (
    "fmt"           // Printing to terminal (like console.log in JS)
    "html/template" // Renders HTML files with dynamic data
    "net/http"      // Built-in web server - no framework needed
    "os"            // File system operations (reading files)
    "strings"       // String manipulation (split, build strings)
)
```

**Why these specific imports?**
- `fmt` — We need to print server startup messages and errors
- `html/template` — Go's built-in templating engine. It safely injects data into HTML (auto-escapes to prevent XSS attacks)
- `net/http` — Go has a production-ready web server built in. No Express, no Flask, no dependencies
- `os` — We read banner files from disk
- `strings` — We split strings and build output efficiently

## Part 2: Data Structures

```go
type PageData struct {
    Text   string   // What the user typed
    Banner string   // Which style they chose (standard/shadow/thinkertoy)
    Result string   // The generated ASCII art
}
```

**What is a struct?** Think of it as a blueprint for a box with labeled compartments. When we send data to our HTML template, we pack everything into this box. The template can then reach into any compartment by name.

**Example in action:**
```
PageData{
    Text:   "Hello",
    Banner: "standard",
    Result: " _    _   ... (ascii art here)",
}
```

The HTML template accesses these with `{{.Text}}`, `{{.Banner}}`, `{{.Result}}`.

```go
var BannerMap = make(map[string][]string)
```

Let me unpack this piece by piece:

- `var` — declaring a variable
- `BannerMap` — the name (capitalized = exported/public, accessible from other packages)
- `map[string][]string` — this is the type. Let me visualize it:

```
BannerMap is a dictionary where:
  KEY (string)  →  VALUE (slice of strings)

  "standard"    →  ["", " _ ", "| |", "|_|", ...]  (each line of the file)
  "shadow"      →  ["", " _ ", "|  ", "|_ ", ...]
  "thinkertoy"  →  ["", " o ", " | ", " o ", ...]
```

**Why `make()`?** In Go, maps must be initialized before use. `make` allocates memory for it. Without `make`, it's `nil`, and writing to a nil map causes a runtime panic (crash).

## Part 3: The main() Function

```go
func main() {
    // Load all banner files at startup
    banners := []string{"standard", "shadow", "thinkertoy"}
    for _, name := range banners {
        if err := loadBanner(name); err != nil {
            fmt.Printf("Error loading %s: %v\n", name, err)
            os.Exit(1)
        }
    }
```

**What's happening step by step:**

1. `banners := []string{...}` — Create a slice (dynamic array) of the three banner names
2. `for _, name := range banners` — Loop through each name. The `_` ignores the index (0, 1, 2) because we don't need it. `name` gets each value.
3. `loadBanner(name)` — Call our function to read the file (defined later)
4. `if err != nil` — Go doesn't have try/catch. Instead, functions return errors as values. You check them explicitly. This is a core Go philosophy: **errors are values, handle them immediately**.
5. `os.Exit(1)` — If any banner fails to load, kill the program. Exit code 1 means "something went wrong." Exit code 0 means success.

**Why load at startup?** Reading files is slow. If we read the banner file on every request, 1000 users = 1000 file reads. By loading once at startup, we read 3 files total, ever. This is called **caching**.

```go
    http.HandleFunc("/", homeHandler)
    http.HandleFunc("/ascii-art", asciiArtHandler)
```

**This is routing.** It's saying:
- "When someone visits `/`, run the `homeHandler` function"
- "When someone submits to `/ascii-art`, run the `asciiArtHandler` function"

Think of it like a receptionist: "If you're here for check-in, go to desk A. If you're here for checkout, go to desk B."

```go
    fmt.Println("Server running on http://localhost:8080")
    http.ListenAndServe(":8080", nil)
}
```

`ListenAndServe` does two things:
1. **Listen** — Open port 8080 and wait for connections
2. **Serve** — When a request comes in, match it to a handler and run it

The `nil` means "use the default router" (the one we just set up with `HandleFunc`).

**This call blocks forever.** The server sits and waits. That's why the print statement comes before it — if it came after, it would never execute.

## Part 4: Loading Banner Files

```go
func loadBanner(name string) error {
    data, err := os.ReadFile("banners/" + name + ".txt")
    if err != nil {
        return err
    }
    BannerMap[name] = strings.Split(string(data), "\n")
    return nil
}
```

**Line by line:**

1. `func loadBanner(name string) error` — Takes a banner name, returns an error (or nil if success)
2. `os.ReadFile("banners/" + name + ".txt")` — If name is "standard", reads `banners/standard.txt`. Returns raw bytes and an error
3. `string(data)` — Converts bytes to a string (Go reads files as bytes, not strings)
4. `strings.Split(..., "\n")` — Splits the entire file content by newlines into a slice. Each element = one line of the file
5. `BannerMap[name] = ...` — Store it in our global map for later use
6. `return nil` — nil error means "everything worked fine"

**The file structure looks like this:**
```
project/
├── main.go
├── banners/
│   ├── standard.txt
│   ├── shadow.txt
│   └── thinkertoy.txt
└── templates/
    └── index.html
```

## Part 5: The Home Handler (GET /)

```go
func homeHandler(w http.ResponseWriter, r *http.Request) {
```

Every HTTP handler in Go has this exact signature:
- `w http.ResponseWriter` — The "pen" you use to write the response back to the browser
- `r *http.Request` — The "envelope" containing everything the browser sent you (URL, method, headers, form data)

The `*` means it's a pointer — you're getting a reference to the request, not a copy. This is efficient for large requests.

```go
    if r.URL.Path != "/" {
        http.Error(w, "404 Not Found", http.StatusNotFound)
        return
    }
```

**Why is this needed?** Go's default router is simple. The pattern `"/"` matches EVERYTHING that starts with `/`. So `/about`, `/xyz`, `/anything` would all hit this handler. This guard ensures only exactly `"/"` is accepted. Everything else gets a 404.

```go
    if r.Method != http.MethodGet {
        http.Error(w, "405 Method Not Allowed", http.StatusMethodNotAllowed)
        return
    }
```

The home page should only respond to GET requests (loading the page). If someone sends a POST/PUT/DELETE to `/`, reject it with 405.

**HTTP Status Codes Quick Reference:**
```
200 = OK (success)
400 = Bad Request (you sent garbage)
404 = Not Found (wrong address)
405 = Method Not Allowed (right address, wrong verb)
500 = Internal Server Error (server broke)
```

```go
    tmpl, err := template.ParseFiles("templates/index.html")
    if err != nil {
        http.Error(w, "500 Internal Server Error", http.StatusInternalServerError)
        return
    }
    tmpl.Execute(w, PageData{})
}
```

**Two-step template process:**
1. **Parse** — Read the HTML file and understand the `{{...}}` placeholders
2. **Execute** — Fill in those placeholders with actual data and write the result to `w`

`PageData{}` — We're sending an empty struct. All fields default to empty strings `""`. This means on first load, the form is blank and no result is shown.

## Part 6: The ASCII Art Handler (POST /ascii-art)

```go
func asciiArtHandler(w http.ResponseWriter, r *http.Request) {
    if r.Method != http.MethodPost {
        http.Error(w, "405 Method Not Allowed", http.StatusMethodNotAllowed)
        return
    }
```

This endpoint only accepts POST. When the user clicks "Generate" on the form, the browser sends a POST request with the form data.

```go
    r.ParseForm()
    text := r.FormValue("text")
    banner := r.FormValue("banner")
```

`r.ParseForm()` — Reads the raw form body (`text=Hello&banner=standard`) and parses it into a usable map.

`r.FormValue("text")` — Gets the value of the form field named "text". This matches the `name` attribute in your HTML:
```html
<input name="text" ...>
<select name="banner" ...>
```

**The connection between HTML and Go:**
```
HTML form:  name="text" value="Hello"
     ↓ (browser sends POST)
Go server:  r.FormValue("text") returns "Hello"
```

```go
    if text == "" {
        http.Error(w, "400 Bad Request", http.StatusBadRequest)
        return
    }

    if banner == "" {
        banner = "standard"
    }

    bannerLines, ok := BannerMap[banner]
    if !ok {
        http.Error(w, "404 Not Found", http.StatusNotFound)
        return
    }
```

**Validation in order:**
1. No text? → 400 (you didn't give us anything to work with)
2. No banner selected? → Default to "standard" (be forgiving)
3. Banner name not in our map? → 404 (someone tampered with the form, sent "comic_sans" as the banner)

The `ok` pattern is idiomatic Go. When you read from a map:
```go
value, ok := myMap[key]
// ok = true  → key exists, value is the data
// ok = false → key doesn't exist, value is zero-value
```

```go
    result := render(text, bannerLines)

    tmpl, err := template.ParseFiles("templates/index.html")
    if err != nil {
        http.Error(w, "500 Internal Server Error", http.StatusInternalServerError)
        return
    }

    data := PageData{
        Text:   text,
        Banner: banner,
        Result: result,
    }

    tmpl.Execute(w, data)
}
```

Now we send back the SAME page, but this time `PageData` is filled with data. The template can show:
- The text the user entered (so the input field is pre-filled)
- Which banner was selected (so the dropdown stays on their choice)
- The ASCII art result

## Part 7: The Render Function (The Core Algorithm)

This is the heart of the application. Understanding this requires understanding how banner files work.

### How Banner Files Are Structured

```
Line 0:  (empty)
Line 1:  row 0 of SPACE character (ASCII 32)
Line 2:  row 1 of SPACE
Line 3:  row 2 of SPACE
...
Line 8:  row 7 of SPACE
Line 9:  (empty separator)
Line 10: row 0 of '!' character (ASCII 33)
Line 11: row 1 of '!'
...
Line 17: row 7 of '!'
Line 18: (empty separator)
Line 19: row 0 of '"' character (ASCII 34)
...
```

**The pattern:** Each character takes **9 lines** (8 lines of art + 1 empty separator). The first line (line 0) is empty.

**Formula to find any character:**
```
start_line = (ASCII_value - 32) × 9 + 1
```

Example for 'A' (ASCII 65):
```
start = (65 - 32) × 9 + 1 = 33 × 9 + 1 = 298
Lines 298-305 contain the 8 rows of 'A'
```

### The Render Function Explained

```go
func render(input string, banner []string) string {
    if input == "" {
        return ""
    }
```

Guard clause. Empty input → empty output.

```go
    var result strings.Builder
```

**Why `strings.Builder` instead of `result += "something"`?**

In Go, strings are immutable. Every `+=` creates a brand new string in memory:
```
"H"           → allocate
"He"          → allocate new, copy, discard old
"Hel"         → allocate new, copy, discard old
"Hell"        → allocate new, copy, discard old
```

`strings.Builder` uses a growing buffer internally. It's like writing into a notebook versus rewriting the entire notebook every time you add a word. For large ASCII art output, this makes a real performance difference.

```go
    parts := strings.Split(input, "\\n")
```

**Critical detail:** This splits on the LITERAL characters `\n` (backslash followed by n), NOT an actual newline. When a user types `Hello\nWorld` in the form, it arrives as the literal string `Hello\nWorld` (6 + 1 + 5 = 12 characters). We split it into `["Hello", "World"]`.

This gives us multi-line support: each part gets rendered as a separate block of ASCII art.

```go
    for i, part := range parts {
        if part == "" {
            if i < len(parts)-1 {
                result.WriteString("\n")
            }
            continue
        }
```

**Handling empty parts (consecutive `\n\n`):**

If input is `"Hi\\n\\nWorld"`, parts = `["Hi", "", "World"]`.

The empty string in the middle means "blank line." We write a newline for it.

The check `i < len(parts)-1` prevents adding a trailing newline after the very last part. It's about clean output — no extra blank line at the end.

`continue` — Skip the rest of the loop body, go to next iteration.

```go
        for row := 0; row < 8; row++ {
```

Each ASCII art character is 8 rows tall. We build the output **row by row across all characters**, not character by character.

Here's why. If we have "AB", we DON'T want:

```
All 8 rows of A
then all 8 rows of B
```

We WANT:

```
Row 0 of A + Row 0 of B
Row 1 of A + Row 1 of B
Row 2 of A + Row 2 of B
...
Row 7 of A + Row 7 of B
```

This is how you get characters side by side.

**Visualized:**
```
Row 0:   _     ____
Row 1:  / \   |  _ \
Row 2: / _ \  | |_) |
Row 3:/ ___ \ |  _ <
...
         A        B       ← side by side!
```

```go
            for _, ch := range part {
                if ch < 32 || ch > 126 {
                    ch = ' '
                }
```

For each character in the current word:
- `range part` iterates through each character (rune) in the string
- `_` ignores the index
- `ch` is the character (type `rune`, which is Go's name for a Unicode code point — essentially an int32)
- If the character is outside printable ASCII range (32-126), replace it with space. This prevents crashes from weird input.

```go
                start := (int(ch) - 32) * 9 + 1
```

**The formula in action.** Let's trace 'H' (ASCII 72):
```
start = (72 - 32) * 9 + 1
start = 40 * 9 + 1
start = 361
```

So lines 361-368 in the banner file contain the 8 rows of 'H'.

`int(ch)` converts the rune to an integer (its ASCII value).

```go
                if start+row < len(banner) {
                    result.WriteString(banner[start+row])
                }
            }
            result.WriteString("\n")
        }
    }

    return result.String()
}
```

- `banner[start+row]` — Gets the specific line from the banner file. For row 0 of 'H', it's `banner[361]`. For row 3 of 'H', it's `banner[364]`.
- `start+row < len(banner)` — Bounds check to prevent index-out-of-range panic
- After all characters in a row are written, add a newline (`\n`) to move to the next row
- `result.String()` — Convert the Builder's internal buffer to a regular string for return

## Complete Request Flow

Let me trace a full request to cement understanding:

```
1. User opens browser → GET /
2. homeHandler runs → serves empty form
3. User types "Hi", selects "shadow", clicks Generate
4. Browser sends → POST /ascii-art with body: text=Hi&banner=shadow
5. asciiArtHandler runs:
   a. Validates method (POST ✓)
   b. Parses form (text="Hi", banner="shadow")
   c. Validates text not empty ✓
   d. Looks up BannerMap["shadow"] ✓
   e. Calls render("Hi", shadowLines)
   f. render processes:
      - Split "Hi" by "\\n" → ["Hi"]
      - For part "Hi":
        - Row 0: get shadow line for 'H' row 0 + shadow line for 'i' row 0
        - Row 1: get shadow line for 'H' row 1 + shadow line for 'i' row 1
        - ... (8 rows total)
   g. Returns ASCII art string
   h. Parses template, fills PageData, sends HTML response
6. Browser displays page with ASCII art result
```

## Key Concepts to Remember for Building Similar Projects

**1. Go web server pattern:**
```go
http.HandleFunc("/path", handlerFunction)
http.ListenAndServe(":port", nil)
```

**2. Handler signature — always the same:**
```go
func name(w http.ResponseWriter, r *http.Request)
```

**3. Template flow — always two steps:**
```go
tmpl, err := template.ParseFiles("file.html")
tmpl.Execute(w, data)
```

**4. Form processing — always three steps:**
```go
r.ParseForm()
value := r.FormValue("fieldname")
// validate value
```

**5. Error handling — always check immediately:**
```go
result, err := someFunction()
if err != nil {
    // handle it RIGHT HERE
    return
}
// safe to use result
```

**6. The banner math formula:**
```
line = (ASCII_value - 32) × 9 + 1 + row_number
```

This project is a clean example of the **MVC pattern** in Go: the struct is the Model, the template is the View, and the handlers are the Controllers.
